# 为什么需要打包? √

随着项目的越来越大和复杂,大型项目当中含有多个js文件,这时候就需要考虑多个js文件之间的加载顺序和文件之间的各种依赖关系,这给开发者带来了诸多不便,并且多个文件分开加载会带来一定的性能损耗,出于这一目的的考虑,打包应用程序应运而生

逻辑多,文件多,项目复杂度提高了

前端技术的不断发展,是因为前端项目的复杂度不断提高推动的,比如说 想要想要前端代码具有类型校验能力,出现了typescript,觉得css不好用,于是出现了各种css预处理器

针对前端项目复杂度提升带来的一个项目需要依赖多个模块的场景,诞生了各种模块化的方案,其中webpack是比较具有代表性的打包工具

# AMD

AMD规范使用define函数进行定义模块 该函数接收三个参数 ,

第一个参数'getSum'表示给模块起的名字

第二个参数表示当前模块需要的依赖 第三个参数,可以是一个函数也可以是一个对象

如果是函数,可以利用函数的返回值将定义的模块接口导出 如果是对象,这个对象本身就是当前模块的导出

# 通过AMD规范定义模块的好处是

它显式的表达出了每个模块所依赖的其他模块有哪些,并且模块的定义,也不在绑定在全局对象window上,这更进一步增强了模块的安全性,不必担心模块在其他地方被篡改

# COMMOMJS

2009年被提出的js规范

最开始是为了定义服务端的模块标准,而不适用于浏览器环境,而后nodejs采用并实现了部分规范,在模块系统上做了一些调整,一般来说,不会严格区分COMMOMJS和nodejs的模块标准,两种标准之间近乎一致,因此在语境当中,nodejs使用的模块化标准一般就指代COMMOMJS

在COMMOMJS当中,一个文件就是一个模块,并且拥有属于自己的独立作用域和上下文,模块的依赖,通过**require()**进行引入,

模块的接口暴露使用export导出

# AMD和COMMOMJS的共同点

AMD和COMMOMJS共同之处就是强调了模块的依赖必须显示引入,这样方便在维护复杂模块时,开发者可以不比关心各个模块之间引入顺序的问题

# 3.1 loader

为什么需要loader，loader是什么?

定义，loader 就是一个 node 模块，接收一些参数，将不被webpack识别的资源转化成webpack能够识别的模块

loader让webpack能够去处理那些非js文件（webpack自身只能够处理js和JSON）

loader 就是一个 node 模块，这很符合 webpack 中「万物皆模块」的思路。既然是 node 模块，那就一定会导出点什么。在 webpack 的定义中，loader 导出一个函数，loader 会在转换源模块（resource）的时候调用该函数。在这个函数内部，我们可以通过传入 `this` 上下文给 Loader API 来使用它们。回顾一下头图左边的那些模块，他们就是所谓的源模块，会被 loader 转化为右边的通用文件，因此我们也可以概括一下 loader 的功能：把源模块转换成通用模块。

# 3.1.8 webpack5 使用资源模块类型打包处理图片文件

**目标:**使用asset module方式 (webpack5版本新增的)

资源模块(asset module)是一种模块类型，它允许使用资源文件（字体，图标等）而无需配置额外 loader。

即webpack5当中不配置loader 一样可以将图片进行打包且不报错

在 webpack 5 之前，通常使用：

  • raw-loader 将文件导入为字符串
  • url-loader 将文件作为 data URI 内联到 bundle 中=>转base64
  • file-loader 将文件发送到输出目录

资源模块类型(asset module type)，通过添加 4 种新的模块类型，来替换所有这些 loader：

超过8kb的直接打包输出到imgage文件夹当中

资源模块类型(asset module type)，通过添加 4 种新的模块类型，来替换所有这些 loader：

  • asset/resource 发送一个单独的文件并导出 URL。之前通过使用 file-loader 实现。
  • asset/inline 导出一个资源的 data URI。之前通过使用 url-loader 实现。
  • asset/source 导出资源的源代码。之前通过使用 raw-loader 实现。
  • asset 在导出一个 data URI 和发送一个单独的文件之间自动选择。之前通过使用 url-loader，并且配置资源体积限制实现。
当在 webpack 5 中使用旧的 assets loader（如 `file-loader`/`url-loader`/`raw-loader` 等）和 asset 模块时，你可能想停止当前 asset 模块的处理，并再次启动处理，这可能会导致 asset 重复，你可以通过将 asset 模块的类型设置为 `'javascript/auto'` 来解决。

  1. url-loader可将文件转换为 base64 URI

    import img from './image.png'
url-loader 的工作方式与 file-loader 类似，但如果文件小于字节限制，则可以返回 DataURL。

// webpack.config.js
module.exports = {
  module: {
    rules: [
		            {
                //使用url-loader 需要依赖于file-loader
                //webpack4处理图片资源   问题:=>这种方式处理不了html中img图片 地址无法解析
                test: /\.(png|jpg|gif|jpeg|webp)$/i,
                loader: 'url-loader',
                options: {
                    limit: 8 * 1024, //图片小于8kb的时候 就会被base64处理
                    //    base64会将图片转换成base64编码的方式,变成字符串,浏览器可以将该字符串当成图像的内容进行解析
                    //    优点,减少页面请求数量,减轻服务器压力
                    //    缺点, 图片体积增大约33%(文件请求速度更慢)
                    //    问题 因为url-loader默认使用es6模块化解析 而html-loader引入的是commonjs
                    //    解析时候会报错
                    //    解决方法 =>关闭url-loaderd的es6模块化,使用commonjs解析
                    esModule: false, //关闭url-loaderd的es6模块化,使用commonjs解析
                 //图片重命名   
                    name: '[hash:10].[ext]', //给图片进行重命名 [hash:10] 
                    //取图片的hash的前十位 [ext]取文件原来拓展名
                },
                type: 'javascript/auto',    //webpack5当中使用webpack4的loader 需要添加这一项
            }, // webpack4当中使用loader处理图片资源
            {
                //处理html当中的img图片,负责引入img 从而能被url-loader处理
                test: /\.html$/i,
                loader: 'html-loader'
            },
        ]
      }
    ]
  }
}

# webpack-dev-server

使用webpack-dev-server启动一个快捷的本地开发服务器时会采用config.devServer参数，和生产环境不同，开发环境下webpack不会将执行文件打包到本地文件夹，而是存放到内存中，devServer.publicPath的含义和output.publicPath完全不同，它不会修改html中资源的路径，有点类似于output.path，用于配置前端文件的服务器访问路径。

  • 必须以/开头，以/结尾
  • 如果配置了一个相对路径，会导致服务器无法访问到文件，出现空白页面
devServer: {
    publicPath: '/assets/'
}
